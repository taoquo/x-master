# Bookmark Detail Drawer Design

## Goal

将当前常驻右侧详情栏改为“按需出现的轻量右侧浮动抽屉”，减少列表页被长期占用的宽度，同时保留书签浏览过程中的详情查看能力。

要求：

- 默认展示纯列表态，不显示详情栏
- 点击卡片后，右侧轻抽屉直接出现
- 抽屉为浮层，不推挤主列表，不触发瀑布流重排
- 抽屉关闭时取消当前选中状态
- 抽屉定位为“内容阅读面板”，不是工具面板
- 抽屉外的更深一层只保留大图查看和视频播放

## Current Problem

当前右侧详情区是固定三栏布局的一部分。

这会带来 3 个问题：

- 未选中任何书签时，右栏长期空置，占用可用宽度
- 详情区常驻，主列表始终被压缩，降低列表浏览效率
- 详情态没有明确退出机制，无法轻易回到纯浏览态

这不是单纯的视觉问题，而是信息架构问题。

列表页的主画布应该优先服务：

- 搜索
- 筛选
- 批量浏览卡片

详情应该是按需调用的上下文层，而不是永久占据布局的一列。

## Design Summary

将详情区从“固定三栏右栏”改为“轻量右侧浮动抽屉”。

核心变化：

- 默认无详情栏，只有列表
- 点击卡片后直接打开右侧抽屉，不增加额外“查看详情”步骤
- 抽屉悬浮在页面右侧，不推挤主内容
- 关闭抽屉时同步取消卡片选中

抽屉承担完整详情职责。

只有两类内容继续保留在抽屉外的更深层：

- 大图查看
- 视频播放

## Why Not Other Approaches

### Not Fixed Sidebar

不继续使用固定右栏。

原因：

- 长期占宽
- 空态浪费明显
- 纯浏览态和详情态边界不清晰

### Not Push Drawer

不采用“打开后推挤主内容”的抽屉。

原因：

- 会压缩瀑布流列表宽度
- 卡片布局会重排
- 用户视线和滚动上下文容易跳动

### Not Heavy Modal

不采用重遮罩的居中 modal 作为主详情方式。

原因：

- 打断列表浏览流
- 与当前“边浏览边判断”的场景不匹配
- 更适合作为媒体沉浸层，而不是常规详情层

## Interaction Model

### Default State

默认状态为：

- 无选中卡片
- 抽屉关闭
- 页面是纯列表浏览态

### Open Drawer

用户点击任意卡片时：

1. 当前卡片进入选中态
2. 右侧轻抽屉直接打开
3. 抽屉展示该书签的完整详情

不增加二次点击“查看详情”的步骤。

原因：

- 点击卡片本身就天然表达“查看这条内容”
- 二次动作只会增加摩擦

### Switch Selection

当抽屉已经打开时，点击另一张卡片：

- 不关闭抽屉
- 直接切换抽屉内容
- 同步更新选中态

### Close Drawer

关闭抽屉时：

- 抽屉消失
- 当前选中卡片取消选中
- 页面回到纯列表态

这意味着系统里不存在“选中了但抽屉没开”的持久状态。

这样可以保持状态模型简单稳定。

### Close Actions

支持的关闭方式：

- 抽屉右上角关闭按钮
- `Esc`

不依赖重遮罩，不把点击空白区域作为主要关闭方式。

### Media Layer

从抽屉内点击媒体时：

- 进入媒体沉浸层
- 支持大图查看或视频播放

关闭媒体层后：

- 回到原抽屉
- 保留当前书签上下文

## Layout Behavior

### Drawer Placement

抽屉贴靠页面右侧，以浮层形式存在。

要求：

- 不改变主列表容器宽度
- 不触发列表重排
- 不影响当前滚动位置

### Overlay Style

抽屉是轻量浮层，不使用重遮罩。

表现上可以有：

- 轻阴影
- 轻边界
- 轻入场动画

但不应让页面其他区域进入“被锁住”的重模态状态。

### Recommended Width

桌面端建议宽度：

- `440px - 520px`

目标是在可读性和不压迫主列表之间取得平衡。

## Content Strategy

抽屉被定义为“内容阅读面板”。

这意味着它的首要任务是：

- 让用户快速读懂这条书签是什么
- 然后顺手完成少量高频操作

而不是：

- 把所有字段堆成一个工具管理栏

### Primary Content

抽屉内保留完整详情内容，包括：

- 作者信息
- 时间信息
- 正文内容
- 媒体预览
- `Open on X`
- 标签与加标签能力
- 其他 metadata

### Secondary Layer

抽屉外的更深层只保留：

- 大图查看
- 视频播放

不再引入额外的“完整详情页”或第二层文字详情弹窗。

## Visual Hierarchy

当前固定右栏的结构更接近：

- `Metadata`
- `Summary`
- `Media`
- `Tags`

这会让正文内容退居次位。

新的轻抽屉应改成：

- `Header`
- `Content`
- `Media`
- `Actions`
- `Secondary metadata`

### Recommended Sections

#### 1. Header

顶部放：

- 头像
- 作者名
- `@handle`
- 时间
- 关闭按钮

Header 负责建立“这是谁、什么时候”的基础上下文。

#### 2. Content

正文区作为主视觉中心。

要求：

- 版式更像阅读内容，而不是元数据块
- 行高、段距更适合阅读
- 不要被一堆 section label 打碎

#### 3. Media

媒体预览尺寸要比当前固定右栏更大。

点击进入媒体沉浸层。

#### 4. Actions

高频动作集中放在正文和媒体之后：

- `Open on X`
- 当前标签
- 添加标签

#### 5. Secondary Metadata

低频 metadata 保留，但降级展示。

原则：

- 不能压过正文
- 不能放在最顶层成为第一关注点

## Motion

建议使用轻量过渡：

- 从右侧短距离 slide in
- 同时配合轻微 fade

推荐时长：

- `180ms - 220ms`

目标是轻、快、明确，不做厚重 modal 动画。

## State Rules

状态简化为两种主态：

1. 纯浏览态
2. 浏览 + 详情抽屉态

状态约束：

- 抽屉打开时一定有选中项
- 抽屉关闭时一定没有选中项

这样 `selectedBookmarkId` 和 `drawerOpen` 不会长期脱钩。

## Non-Goals

本次不做：

- 常驻三栏布局的保留模式
- push 式抽屉
- 重遮罩 modal 作为主详情容器
- 额外的长文详情页
- 详情关闭后保留选中高亮

## Risks

### Too Much Content in Drawer

如果把过多低频信息塞进抽屉，抽屉会重新变成现在的“右侧工具栏”。

缓解方式：

- 让正文优先
- 弱化低频 metadata
- 保持操作数量克制

### Overlay Too Weak

如果抽屉视觉上过轻，用户可能感知不到它已经进入详情态。

缓解方式：

- 使用清晰的边界和阴影
- 卡片保留选中态
- 打开关闭动画保持明确

### Mobile / Narrow Width Adaptation

窄屏下右侧轻抽屉空间不足。

当前设计优先针对桌面端。

后续可按视口宽度降级为：

- 更窄抽屉
- 或全宽覆盖面板

但这不属于本次桌面端设计主决策。

## Testing Strategy

至少覆盖：

- 默认状态不展示详情抽屉
- 点击卡片后打开抽屉并显示对应内容
- 点击另一张卡片时切换抽屉内容
- 关闭抽屉后取消选中状态
- `Esc` 可关闭抽屉
- 打开媒体层后关闭，仍回到原抽屉
- 抽屉打开时主列表容器宽度不发生重排式变化

## Implementation Order

1. 先将详情区从固定三栏中解耦成可控抽屉状态
2. 替换点击卡片后的交互：从“右栏同步展示”改为“打开抽屉”
3. 重构详情抽屉的内容层级，提升正文优先级
4. 保留并接通现有媒体沉浸层
5. 补状态和交互测试
